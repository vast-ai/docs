Picking the right GPU mix for a serverless workload is a performance-per-dollar problem, not a spec-sheet problem. A cheaper GPU is only a win if it can keep up with your workload, and a faster GPU is only a win if the extra throughput pays for the extra hourly rate. Without measured numbers on your actual template, the only ways to find the optimum are guesswork or production trial-and-error.

Vast Serverless solves this in two complementary ways:

- **Automatically**, while your Workergroup runs. The serverless engine continuously benchmarks workers and uses the results to decide which to recruit, route to, and release.
- **On-demand**, before you commit. The `vastai run benchmark` CLI command rents one instance per GPU class, runs the same benchmark workload, and reports performance per dollar so you can compare options up front.

Both mechanisms exercise the same workload: the benchmark that the template's PyWorker already has baked in. You don't need to write or modify it; pre-built templates (vLLM, TGI, ComfyUI, etc.) ship with one. If you're authoring a new template, see [Creating PyWorkers](/guides/serverless/creating-new-pyworkers).

## Automatic GPU Selection

When a new Workergroup is created, the serverless engine enters a **learning phase**. During this phase, it recruits a variety of machine types from those allowed by your `search_params`. Each new worker runs the template's benchmark and reports `measured_perf` back to the engine.

As traffic scales up and down, the engine builds an **application-specific model of cost vs. performance**, which it then uses to make informed decisions about future worker recruitment, request routing, and worker release. The result is that, over time, your Workergroup converges on the cheapest mix of GPUs that can actually serve your traffic, without you having to hand-pick them.

For tips on accelerating that convergence in the first day of operation, see [Managing Scale → Initial Rollout](/guides/serverless/managing-scale).

## On-Demand Benchmarking with the CLI

The `vastai run benchmark` command lets you do the same evaluation yourself, before (or independent of) any Workergroup. You give it a template and a list of GPU classes; for each GPU, the CLI rents one instance, runs the template's benchmark workload, records the measured performance, looks up the rental's hourly price, and reports performance per dollar. Multiple GPUs run in parallel and tear themselves down when finished.

<Note>
  This is a CLI-only tool today. There is no SDK equivalent in the `vastai` Python package. Run it from your terminal or a CI job; consume the results via `--raw` for scripting.
</Note>

### When to Use It

- Choosing the GPU classes to allow in a new Workergroup's `search_params`.
- Comparing two templates (for example, vLLM vs. TGI for the same model) on the same hardware.
- Validating that a template fits and runs correctly on a GPU before allowing it into production.
- Producing a perf/dollar number for capacity planning or budget conversations.

### Prerequisites

- A Vast.ai account with a positive balance. Each benchmark run rents real instances; budget accordingly.
- An API key configured (see [Authentication](/cli/authentication)).
- The template's hash or ID. You can find these on the [Templates dashboard](https://cloud.vast.ai/templates/) or via `vastai search templates`.

### Quickstart

Benchmark a template against the default GPU set (RTX 5090, RTX 4090, RTX 3090, RTX A6000):

```bash
vastai run benchmark --template_hash 79ebdd2ebfb9d42cedf7a221c42d37a5
```

Benchmark a template against an explicit list of GPUs:

```bash
vastai run benchmark \
  --template_hash 393fa8572e6c73c927c8275fe4dffd53 \
  --gpus RTX_4090,RTX_3090
```

Use the `Nx` prefix to request multi-GPU configurations on a per-token basis:

```bash
vastai run benchmark \
  --template_hash 40ef49becc953aa910ee05bd4653b9b3 \
  --gpus "2x RTX_4090, 2x RTX_3090"
```

The CLI prints a confirmation prompt with the worst-case cost before any rental is created. Pass `-y` to skip it in scripts.

### How It Works

For each GPU in your list, the runner:

1. **Resolves the GPU count.** If a token has an `Nx` prefix, that wins. Otherwise, if `--num_gpus` is set, it is used. Otherwise, the runner auto-sizes from the template's `gpu_total_ram` filter so VRAM fits.
2. **Pre-flights the marketplace.** Counts matching offers using the template's `extra_filters` plus the GPU and `num_gpus`. GPUs with zero matching offers are skipped, and the CLI explains which filter excluded them.
3. **Creates an ephemeral endpoint and Workergroup.** Each GPU gets its own scratch endpoint named `benchmark Nx <GPU> (<uuid>)`, scoped to one worker (`cold_workers=1`, `max_workers=1`).
4. **Polls for an idle worker.** The PyWorker boots, runs the template's built-in benchmark workload, and reports `measured_perf`. The runner waits for `status=idle` and `measured_perf > 0`.
5. **Reads the rental price.** Once the worker reaches idle, the CLI fetches the instance's `dph_total` (the hourly rate Vast actually charged for that contract) so the perf/dollar ratio reflects the real run, not a marketplace estimate.
6. **Records the result and tears down.** The Workergroup is deleted first, then the endpoint. An `atexit` handler sweeps anything left if the process dies unexpectedly.

The rentals run in parallel, with a small stagger between submits to stay under the autoscaler's create rate limit.

### Reading the Output

While the run is in progress, the CLI renders a live table:

| Column | Meaning |
|--------|---------|
| **GPU** | The GPU class being benchmarked. |
| **Status** | Current worker state: `provisioning`, `waiting_for_worker`, `loading`, `idle`, `done`, `failed`, `timeout`, `no_worker`, `skipped`. |
| **Endpoint** | The ephemeral endpoint ID created for this run. |
| **Worker** | The worker (instance) ID, once one has been recruited. |
| **Elapsed** | Time since the worker started running. Freezes on terminal status. |
| **Perf** | Measured performance from the template's benchmark workload. Units are template-specific (tokens/second for LLMs, etc.). |
| **$/hr** | The rented worker's `dph_total`, the hourly rate the contract is being billed at. |
| **Perf/$/hr** | Cost-efficiency score: measured performance divided by hourly price. Higher is better. |

After the run, the CLI prints a sorted summary by `Perf/$/hr`, so the most cost-efficient GPU for your workload is at the top. With `--raw`, the same data is emitted as JSON for scripting:

```json
[
  {
    "gpu_name": "RTX 4090",
    "rental_dph": 0.428,
    "measured_perf": 142.3,
    "status": "ok",
    "perf_per_dollar": 332.5
  }
]
```

### Cost and Timeout

Each GPU rental runs for up to `--timeout` seconds (default 3600). The pre-flight estimate shown in the confirmation prompt is a worst-case upper bound: `sum(median_price * timeout_hours)` across all GPUs. Real runs almost always finish well before the ceiling because the runner exits as soon as it reads a valid `measured_perf`.

You can tighten the ceiling for cheaper runs:

```bash
vastai run benchmark \
  --template_hash 393fa8572e6c73c927c8275fe4dffd53 \
  --timeout 600 -y
```

<Warning>
  If you `Ctrl+C` mid-run, the CLI installs a SIGINT handler that finishes deleting in-flight endpoints and Workergroups before exiting. Do not kill the process again during cleanup, otherwise you may need to manually remove the leftovers with `vastai delete endpoint` and `vastai delete workergroup`.
</Warning>

### Common Outcomes

| Status | What it means | What to do |
|--------|---------------|------------|
| `ok` | Benchmark completed and reported `measured_perf`. | Use the result. |
| `skipped` | No marketplace offer matched after applying the template's filters. | The CLI prints which filter blocked the GPU. Loosen `extra_filters` on the template, or pick a different GPU. |
| `no_worker` | The autoscaler did not rent any instance within 120 seconds. | Often a scoring or template+GPU mismatch the pre-flight missed. Try a different GPU or relax filters. |
| `failed` | Workers reached a terminal state (`stopped`, `destroying`, `unavail`) without ever becoming idle. | Inspect worker logs in the dashboard. Common causes are model download failure or an OOM during load. |
| `timeout` | The worker was still loading or running when `--timeout` elapsed. | Increase `--timeout`, or check whether the host is unusually slow (Docker pull stalls are the typical culprit). |
